01

Foundations — What CRUD Actually Means

Beginner

Almost every application you use — Instagram, your to-do list, your bank's dashboard — is, underneath the UI, just four operations happening to records of data. That's CRUD: Create, Read, Update, Delete.

The four operations

OperationMeaningEveryday exampleSQL equivalent
CreateAdd a new recordPosting a new photoINSERT
ReadFetch existing record(s)Opening your feedSELECT
UpdateModify an existing recordEditing a captionUPDATE
DeleteRemove a recordDeleting a photoDELETE

Every "feature" you can imagine in an app is really just one of these four things applied to some kind of record — a user, a photo, an order, a task.

The client-server-database triangle

Three actors, three jobs:

Analogy: a restaurant

You (client) tell the waiter (server) what you want. The waiter never cooks — they take your order, check it makes sense, and walk it to the kitchen (database). The kitchen prepares or retrieves the food and hands it back to the waiter, who brings it to your table. You never talk to the kitchen directly, and the kitchen never talks to you directly — the waiter is the only one that touches both sides. That's exactly the server's job between client and database.

Common misconception
"The frontend talks directly to the database." In a real app, it almost never does — for security and control, the client only ever talks to the server's API, and only the server is allowed to query the database.
Quick exercise

You're building a "Notes" app. A user clicks "Save" on a new note. Which CRUD operation is this, and which actor initiates it?

🔑 Key takeaways

  • CRUD = Create, Read, Update, Delete — the four things you can do to any record.
  • The client renders UI and sends requests; the server enforces logic; the database persists data.
  • The client should never touch the database directly — the server is the gatekeeper.

Speaking REST — Designing the API

Beginner–Intermediate

Now that we know client, server, and database each have a job, we need a shared language for the client and server to talk in. That language is usually REST — a convention for mapping CRUD operations onto URLs and HTTP methods.

HTTP verbs map directly to CRUD

CRUDHTTP VerbExample endpointMeaning
CreatePOSTPOST /tasksCreate a new task
ReadGETGET /tasks / GET /tasks/12List all, or fetch one
UpdatePUT / PATCHPUT /tasks/12Replace or modify task 12
DeleteDELETEDELETE /tasks/12Remove task 12

Anatomy of a request & response

// Client sends: POST /tasks HTTP/1.1 Content-Type: application/json { "title": "Buy milk", "done": false } // Server responds: HTTP/1.1 201 Created Content-Type: application/json { "id": 42, "title": "Buy milk", "done": false }

Notice the request has a body (the new data) and the response has a status code (201 = "successfully created") plus the saved record, now including a server-assigned id.

Status codes worth knowing

CodeMeaningWhen you'll see it
200 OKSuccessGET, PUT, DELETE succeeded
201 CreatedSuccess, new resourceAfter a POST
400 Bad RequestClient sent invalid dataMissing required field
404 Not FoundResource doesn't existGET /tasks/999 when it's gone
500 Server ErrorSomething broke server-sideDatabase connection failed
✓ Good REST design
  • GET /tasks/12
  • Nouns in the URL, verbs in the HTTP method
  • Consistent, predictable status codes
✗ Common mistake
  • GET /getTask?id=12
  • Verbs baked into the URL itself
  • Everything returns 200, even errors
Common misconception
"PUT and PATCH are the same thing." PUT typically replaces the whole record; PATCH updates only the fields you send. Mixing them up can silently wipe fields you didn't mean to touch.
Quick exercise

You need an endpoint to permanently remove task #7. Write the HTTP method + URL.

🔑 Key takeaways

  • REST maps CRUD to HTTP verbs: POST, GET, PUT/PATCH, DELETE.
  • URLs should be nouns (resources); the verb lives in the HTTP method.
  • Status codes tell the client what happened without parsing the body.

03

Building the App — From Request to Row

Intermediate

Let's trace one full round-trip for a real feature: marking a task as done in a Task Manager app. This is where the pieces click together.

Step-by-step trace: PATCH /tasks/12

1. CLIENT — user checks the box in the UI onCheck() → fetch("/tasks/12", { method: "PATCH", body: { done: true } }) 2. SERVER — receives the request - Checks the task exists - Validates "done" is a boolean - Calls the database layer 3. DATABASE — executes the change UPDATE tasks SET done = true WHERE id = 12; 4. SERVER — formats the response HTTP/1.1 200 OK { id:12, title:"Buy milk", done:true } 5. CLIENT — updates the UI from the response setTasks(prev ⇒ prev.map(t ⇒ t.id===12 ? updatedTask : t))

Five hops, one round trip. Every button click in a data-driven app is a version of this same sequence.

Where does "state" actually live?

A subtlety that trips up beginners: the database is the source of truth. What the client shows is just a temporary copy. If two people edit the same task at once, whoever's write reaches the database last "wins" unless the app adds extra logic to prevent that.

Read (GET)
  • Safe — never changes data
  • Can be cached
  • Usually no request body
Write (POST/PUT/DELETE)
  • Changes data — not safe to repeat blindly
  • Should never be cached
  • Usually needs validation
Common misconception
"Updating the UI is the same as saving the data." Changing what's on screen is instant and local — it means nothing until the server confirms the database write succeeded. That's why good apps show a loading or "saving…" state.
Quick exercise

A user deletes a task, but their wifi drops before the request reaches the server. What should the UI do?

🔑 Key takeaways

  • Every interaction is a 5-step round trip: client → server → database → server → client.
  • The database is the single source of truth, not the browser.
  • Reads are safe to repeat; writes are not — treat them differently.

04

Mastery — Validation, Errors & Real Architecture

Advanced

Production CRUD apps aren't just the happy path. Mastery means handling everything that can go wrong, and structuring the app so it stays maintainable.

Validation: never trust the client

Even if your frontend form requires a title, someone can call your API directly with a tool like curl and skip the form entirely. Validation must happen on the server, not just in the browser.

// server-side validation, pseudocode function createTask(body) { if (!body.title || body.title.trim() === "") { return { status: 400, error: "title is required" }; } // only now is it safe to insert return db.insert("tasks", body); }

Handling failure gracefully

FailureWhere caughtGood response
Missing fieldServer validation400 + which field is wrong
Record not foundServer, before DB call404 + clear message
Database offlineServer, DB layer500 + retry-safe message
Network dropClientRevert optimistic UI, offer retry

Analogy: guardrails on a bridge

Validation and error handling are like guardrails on a bridge — drivers (requests) rarely need them, but the one time a request swerves (bad data, dropped connection, a bug), the guardrail is what stops it from crashing the whole system instead of just that one request.

Optimistic UI
  • Updates screen immediately
  • Feels instant
  • Must be able to roll back on failure
Pessimistic UI
  • Waits for server confirmation
  • Feels slower but always accurate
  • No rollback logic needed
Common misconception
"If the server didn't crash, the request succeeded." A server can respond with a perfectly valid HTTP response that still carries an error status (400/404/500). Always check the status code, not just "did I get a response."
Quick exercise

Someone calls DELETE /tasks/999 but task 999 doesn't exist. What status code should the server return?

🔑 Key takeaways

  • Always validate on the server — the client can be bypassed entirely.
  • Every failure mode (missing data, missing record, dead DB, dropped network) needs a defined response.
  • Optimistic UI needs rollback logic; pessimistic UI trades speed for simplicity.

📝 Summary Notes

CRUD is four operations (Create, Read, Update, Delete) mapped to HTTP verbs (POST, GET, PUT/PATCH, DELETE) and SQL commands (INSERT, SELECT, UPDATE, DELETE). The client sends requests, the server validates and applies logic, the database persists the source of truth. Every user interaction is a round trip through all three. Production apps add validation, status codes, and rollback logic to handle the real world, where requests fail.
